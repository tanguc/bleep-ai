# secrets-patterns-db YAML → Internal Schema Field Mapping

**Source files:**
- `rules/vendor/secrets-patterns-db/rules-stable.yml` (secrets: API keys, tokens, credentials)
- `rules/vendor/secrets-patterns-db/pii-stable.yml` (PII: emails, phones, addresses)

**Format:** YAML `patterns: [{pattern: {name, regex, confidence}}]`
**License:** CC-BY 4.0 — attribution required; `source` field in every normalized rule enables this
**Total patterns in vendored files:** 1,610+ in rules-stable.yml; ~30 in pii-stable.yml
**Included in curated manifest:** 8 patterns (see `docs/analysis/CURATED-MANIFEST.md`)

---

## Source Format Reference

Both `rules-stable.yml` and `pii-stable.yml` use the same YAML structure. The top-level key is `patterns:` — a list of objects, each with a single nested `pattern:` key containing three fields.

```
patterns:
  - pattern:
      name: <string>
      regex: <string>
      confidence: low | medium | high
```

Note the nesting: `pattern.name`, `pattern.regex`, `pattern.confidence` — not top-level fields. The outer `pattern:` key has no other purpose; it wraps the three fields.

### Representative Raw YAML Snippet

```yaml
patterns:
  - pattern:
      name: AWS API Key
      regex: AKIA[0-9A-Z]{16}
      confidence: high
  - pattern:
      name: AWS MWS key
      regex: amzn\.mws\.[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}
      confidence: high
```

This is the simplest upstream format of the three vendored sources. Each entry provides exactly three fields and no more.

## ID Generation Rules

secrets-patterns-db patterns have no ID field. IDs must be generated by build.rs using the slugify algorithm applied to `pattern.name`.

### Slugify Algorithm

1. Lowercase all characters
2. Replace spaces and non-alphanumeric characters (except existing hyphens) with a single hyphen
3. Collapse consecutive hyphens to one hyphen
4. Strip leading and trailing hyphens
5. Prefix with `spdb.`

### Examples

| `pattern.name` | Slugified | Final ID |
|----------------|-----------|----------|
| `AWS API Key` | `aws-api-key` | `spdb.aws-api-key` |
| `AWS MWS key` | `aws-mws-key` | `spdb.aws-mws-key` |
| `U.S. Phone Number` | `us-phone-number` | `spdb.us-phone-number` |
| `emails` | `emails` | `spdb.emails` |
| `phones_with_exts` | `phones-with-exts` | `spdb.phones-with-exts` |
| `AWS AppSync GraphQL Key` | `aws-appsync-graphql-key` | `spdb.aws-appsync-graphql-key` |

### Collision Handling

If two patterns produce the same slug (e.g. two patterns named `AWS API Key` in different files), build.rs must detect the collision and append a numeric suffix: `spdb.aws-api-key-2`. In practice this is unlikely because the curated manifest selects at most one pattern per secret type from this source, but the normalization pipeline must be defensive.

## Attribution Compliance

secrets-patterns-db is licensed **CC-BY 4.0** (Creative Commons Attribution 4.0 International). Attribution is required when distributing patterns from this source.

The internal schema enables attribution through the `source` field: every normalized rule from secrets-patterns-db carries `source: secrets-patterns-db`. This field is the mechanism — the actual attribution emission is a build.rs / Phase 3 concern:

- **Recommended approach:** build.rs collects all rules where `source == "secrets-patterns-db"` and writes an attribution notice to a `LICENSE-NOTICES` file at compile time. That file is embedded in the binary as a string constant or included in the distribution package.
- **Minimum compliance:** the `source` field in the binary's embedded YAML is itself an attribution signal, but a human-readable notice in `LICENSE-NOTICES` is preferred.

The `source` field must never be omitted for secrets-patterns-db patterns. Build.rs must validate that every rule parsed from `rules-stable.yml` or `pii-stable.yml` has `source: secrets-patterns-db` set before writing to `rules/patterns.yaml`.

## Low-Confidence Exclusion Policy

secrets-patterns-db `rules-stable.yml` contains 1,610+ patterns. The vast majority are low-confidence or duplicates of patterns already covered by nosey-parker or gitleaks.

**Only INCLUDE decisions from `docs/analysis/CURATED-MANIFEST.md` are normalized.** Build.rs must consult the manifest, not normalize the full file.

The exclusion policy by category:
- `low` confidence patterns from rules-stable.yml: **EXCLUDED** — too many false positives in LLM proxy context
- Patterns already included from nosey-parker or gitleaks: **EXCLUDED** — deduplication rule D-03 from Phase 1
- Patterns for secret types not relevant to proxy-based detection (e.g. infrastructure hostnames): **EXCLUDED**

The 8 included patterns (per curated manifest summary) represent the subset where secrets-patterns-db provides unique value not covered by other sources, typically through its `confidence` field being explicitly set to `high` for well-studied formats.

## RE2-Incompatible Patterns Warning

Phase 1 analysis identified **6 patterns in `pii-stable.yml`** that use regex features unsupported by the Rust `regex` crate (RE2 semantics):
- Lookaheads: `(?=...)` — not supported in RE2
- Lookbehinds: `(?<=...)`, `(?<!...)` — not supported in RE2

These patterns are **EXCLUDED** in the curated manifest and must never reach the internal schema. However, build.rs must include a validation step:

1. For every pattern parsed from secrets-patterns-db files, attempt to compile the regex using the Rust `regex` crate
2. If compilation fails, log a build error identifying the pattern name and reject it
3. Do not silently skip — a compilation failure indicates either an RE2-incompatible pattern that leaked through the manifest filter, or a regex syntax error in the source

This validation is mandatory. The 6 known incompatible patterns (all in `pii-stable.yml`) include phone number patterns that use negative lookbehinds to avoid matching inside larger numbers.

## Before/After Normalization Example

### Source: raw YAML (rules/vendor/secrets-patterns-db/rules-stable.yml)

```yaml
patterns:
  - pattern:
      name: AWS API Key
      regex: AKIA[0-9A-Z]{16}
      confidence: high
```

Fields present in source: `name`, `regex`, `confidence` (all three)
Fields absent from source: `id`, `category`, `subcategory`, `entropy`, `keywords`, `tags`, `checksum_type`, `description`, `severity`, `source`, `replacement_type`

### Normalized internal YAML output (rules/patterns.yaml)

```yaml
- id: spdb.aws-api-key
  name: AWS API Key
  category: secret
  subcategory: aws
  source: secrets-patterns-db
  confidence: high
  regex: 'AKIA[0-9A-Z]{16}'
  keywords: []
  entropy: null
  tags: []
  checksum_type: null
  replacement_type: faker_aws_key
  description: ""
  severity: high
```

**Transform decisions applied:**
- `id`: generated by slugify(`AWS API Key`) = `aws-api-key` → prefixed → `spdb.aws-api-key`
- `name`: `AWS API Key` → copied verbatim
- `regex`: `AKIA[0-9A-Z]{16}` → copied verbatim (RE2-compatible, verified in Phase 1)
- `confidence`: `high` → `high` — direct copy, no transform (this is the only source with explicit confidence)
- `category`: `secret` — assigned from file origin (`rules-stable.yml`) and confirmed by curated manifest
- `subcategory`: `aws` — assigned from curated manifest
- `source`: `secrets-patterns-db` — hardcoded (enables CC-BY 4.0 attribution)
- `entropy`: `null` — no entropy data in source; default disabled
- `keywords`: `[]` — no keywords in source; default empty (regex runs unconditionally)
- `tags`: `[]` — no tags in source; default empty
- `checksum_type`: `null` — `aws` subcategory does not use Luhn
- `replacement_type`: `faker_aws_key` — derived from `secret/aws` per REPLACEMENT-TYPES.md
- `description`: `""` — no description field in source; empty string default
- `severity`: `high` — derived from `confidence: high`

Note: this pattern (`spdb.aws-api-key`) is marked **DUPLICATE** in the curated manifest (np.aws.1 wins for broader prefix coverage). The example above shows what normalization would produce if it were included. The actual `rules/patterns.yaml` will not include this rule.
