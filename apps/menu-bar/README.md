# Bleep menu-bar app

macOS menu-bar dashboard for the bleep gateway. Tray icon shows
today's redaction count. Click → menu with summary + "Open
Dashboard…". Dashboard window has summary cards, category /
top-rules charts, and a live tail of redactions as they happen.

Built with [Tauri 2](https://tauri.app/).

## Architecture

```
menu-bar (Tauri)        gateway (bleep-gateway, sibling Cargo crate)
   │                        │
   │  HTTP poll /stats   ◀──┤  axum server  on /tmp/bleep-stats.port
   │                        │
   │  TCP JSONL events   ◀──┤  event_bus    on /tmp/bleep-events.port
   │                        │
   │  spawns as child    ──▶│  process tree
```

The menu-bar app discovers the gateway via two port files in
`/tmp/`:

- `/tmp/bleep-stats.port` — HTTP `/stats`, `/stats/categories`,
  `/stats/rules`, `/health`
- `/tmp/bleep-events.port` — TCP JSONL stream of `ProxyEvent` (same
  one the TUI consumes)

By default the menu-bar app runs in **observe-only mode** — it
displays whatever a separately-running `bleep-gateway` is doing but
does NOT auto-spawn one. Set `BLEEP_SPAWN_GATEWAY=1` to opt into
auto-spawning; on menu-bar quit, the spawned child is killed
(`kill_on_drop = true`).

## Development

From the repo root:

```bash
# auto-spawn the gateway as a child of the menu-bar app
task menu-bar

# observe-only mode — gateway must already be running (e.g. via `task run`)
task menu-bar:observe

# kill everything and clean port files
task menu-bar:stop
```

`task menu-bar` calls `apps/menu-bar/run-dev.sh`, which:
1. builds `bleep-gateway` in release mode (one-time, fast after)
2. kills any stale instances
3. clears `/tmp/bleep-*.port`
4. runs `cargo run` in `apps/menu-bar/src-tauri/` with `BLEEP_SPAWN_GATEWAY=1`

For a manual debug session:
```bash
cd apps/menu-bar/src-tauri
BLEEP_SPAWN_GATEWAY=1 BLEEP_GATEWAY_BIN=/path/to/bleep-gateway cargo run
```

## Env vars

| Variable | Default | Purpose |
|---|---|---|
| `BLEEP_SPAWN_GATEWAY` | unset | Set to `1`/`true` to make the menu-bar app spawn the gateway as a child process. Without it the app runs in observe-only mode. |
| `BLEEP_GATEWAY_BIN`   | unset | Override the gateway binary path. Only consulted when `BLEEP_SPAWN_GATEWAY` is set. |

## Files

- `src-tauri/src/main.rs` — tray, menu, dashboard window, gateway
  lifecycle, event-bus forwarder, signal handling
- `src-tauri/tauri.conf.json` — app metadata, CSP, withGlobalTauri
- `src-tauri/capabilities/default.json` — Tauri 2 permissions for
  the main window (event listen, webview, window show/hide/focus)
- `src-tauri/icons/` — placeholder PNGs (regenerate before shipping)
- `ui/index.html`, `ui/style.css`, `ui/app.js` — vanilla
  HTML/CSS/JS dashboard. No bundler, no framework.
- `ui/jsconfig.json` — VS Code config; sets up the `@bleep-events/*`
  alias so JSDoc imports resolve to the generated TS types.

## Wire types

The menu-bar's `ProxyEvent` / `RedactedEntry` types come from
`crates/bleep-events/` (shared with the gateway and the TUI binary).
TypeScript declarations are auto-generated by `ts-rs`:

```bash
task bindings
# → writes crates/bleep-events/bindings/{ProxyEvent,RedactedEntry}.ts
```

`ui/app.js` imports them via JSDoc (`/** @typedef {import("@bleep-events/ProxyEvent").ProxyEvent} ProxyEvent */`)
so VS Code autocomplete and type-checking work in the webview JS
without a bundler.

## Tray UX gotchas (do not regress)

- The tray menu is built ONCE at startup. The poller updates the
  dynamic header / status items in place via `MenuItem::set_text`.
  Calling `tray.set_menu(...)` to "refresh" the menu while it's open
  causes macOS to dismiss the open dropdown — feels like the menu
  loses focus to the user.
- `tray.set_title(...)` is safe to call as often as you want.
- The title is text-only ("Bleep · 47"). Setting an icon caused a
  duplicate config-vs-Rust tray entry in an earlier iteration.
- Closing the window via the red traffic light HIDES the window
  rather than quitting (Mac convention for tray-having apps).
  Cmd+Q or "Quit Bleep" actually exits.
